TASK: Stand up the GCP data layer — Cloud SQL (Postgres 16) and a Cloud Storage
bucket — and migrate existing data into them, running ALONGSIDE the live Replit
app. Do NOT change DNS, do NOT point production at GCP, do NOT decommission
anything. The end state is: GCP has a verified copy of the data, ready for a
later cutover. This is an ops runbook (gcloud/gsutil + a one-off copy script),
not an app code change.

PARAMETERS (set these first):
  PROJECT_ID=measure-once-prod          # or your chosen GCP project
  REGION=europe-west2                   # London — UK data residency for PII
  SQL_INSTANCE=measure-once-pg
  DB_NAME=measureonce
  DB_USER=app
  GCS_BUCKET=measure-once-media-<unique-suffix>
Match the Postgres MAJOR version to source: Replit uses Postgres 16, so Cloud SQL
must be POSTGRES_16.

PHASE 0 — Project & APIs
- Create/select the project. Enable: sqladmin, storage, secretmanager,
  run, artifactregistry, compute APIs.
  gcloud config set project $PROJECT_ID
  gcloud services enable sqladmin.googleapis.com storage.googleapis.com \
    secretmanager.googleapis.com run.googleapis.com \
    artifactregistry.googleapis.com compute.googleapis.com

PHASE 1 — Cloud SQL (Postgres 16)
- Create the instance in $REGION (pick tier per expected load; enable automated
  backups and point-in-time recovery):
  gcloud sql instances create $SQL_INSTANCE --database-version=POSTGRES_16 \
    --region=$REGION --tier=db-custom-1-3840 --storage-auto-increase \
    --backup --enable-point-in-time-recovery
- Create DB and app user (store the password in Secret Manager, NOT in shell
  history files):
  gcloud sql databases create $DB_NAME --instance=$SQL_INSTANCE
  gcloud sql users create $DB_USER --instance=$SQL_INSTANCE --password=<gen-strong>
- Prefer connecting via the Cloud SQL Auth Proxy / connector (no public IP). For
  the migration step you may temporarily authorise your IP or run the proxy.

PHASE 2 — Cloud Storage bucket
- Create a uniform-access, private bucket in the SAME region:
  gcloud storage buckets create gs://$GCS_BUCKET --location=$REGION \
    --uniform-bucket-level-access --public-access-prevention
- IMPORTANT: object names must be preserved EXACTLY when copying (the DB stores
  opaque keys like obj:<id>.<ext> that map to object names such as
  design-visit-images/<id>.<ext>, plus the customer-info object paths). Do not
  re-prefix or rename objects, or the stored keys will stop resolving.

PHASE 3 — Service account & IAM (for the app via ADC)
- Create a runtime service account and grant least privilege:
  gcloud iam service-accounts create measure-once-run
  # Cloud SQL client + Storage object admin on the bucket only:
  gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member="serviceAccount:measure-once-run@$PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/cloudsql.client"
  gcloud storage buckets add-iam-policy-binding gs://$GCS_BUCKET \
    --member="serviceAccount:measure-once-run@$PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/storage.objectAdmin"
- App auth uses Application Default Credentials — no key files in the repo.

PHASE 4 — Secret Manager
- Store each app secret as a secret (DATABASE_URL for Cloud SQL, SESSION_SECRET,
  HUBSPOT_TOKEN, GOOGLE_* OAuth, QB_*, GOOGLE_PLACES_API_KEY, the Google/QB token
  encryption keys, etc.). These will be mounted into Cloud Run at cutover.
  Do not commit any secret values.

PHASE 5 — Database migration (Replit Postgres → Cloud SQL)
- Take a full logical dump from the SOURCE (Replit) database. A full pg_dump
  captures ALL tables INCLUDING the migration bookkeeping tables (pgmigrations
  for node-pg-migrate AND public.migrations for the rate-limit package) — both
  must come across or the app will try to re-run migrations on first boot.
    pg_dump "$SOURCE_DATABASE_URL" --no-owner --no-privileges -Fc -f dump.pgc
- Restore into Cloud SQL (via the proxy/connector):
    pg_restore --no-owner --no-privileges -d "$CLOUDSQL_DATABASE_URL" dump.pgc
- Verify row counts match for key tables (users, sessions, design_visits,
  customer_info*, quickbooks tokens, pgmigrations, public.migrations) between
  source and destination. Record the counts.

PHASE 6 — Object storage migration (Replit Object Storage → GCS)
- Reuse the storage abstraction added earlier (storage.js with 'replit' and 'gcs'
  backends). Write a ONE-OFF script scripts/migrate-objects.mjs that:
    * Enumerates every object in the Replit bucket (use the Replit SDK's list()).
    * For each, streams bytes from the Replit backend and writes them to the GCS
      backend under the IDENTICAL object name (no prefix/rename).
    * Is idempotent/resumable (skip objects already present in GCS with the same
      size) and logs a final count + any failures.
  If storage.js lacks a list() capability, add list() to both backends for this
  purpose. Run it with STORAGE creds for both sides available via env/ADC.
- After copy, verify object COUNT and a sample of byte-for-byte checksums match
  between source and GCS.

PHASE 7 — Verification (no cutover)
- Optionally run the containerised app (from the Docker task) LOCALLY pointed at
  Cloud SQL + GCS (STORAGE_BACKEND=gcs, GCS_BUCKET set, Cloud SQL via proxy) and
  smoke-test: log in, open a design visit with photos (signed URLs resolve from
  GCS), load a customer-info submission. Do this against a COPY — be careful not
  to send real customer emails/invoices from a test instance (set SMTP/HubSpot/QB
  to safe/sandbox values or disable side effects).
- Leave Replit production untouched and serving live traffic.

CONSTRAINTS / SAFETY:
- No DNS changes, no production cutover, no decommissioning in this task.
- Source data is live customer PII — keep dumps encrypted, delete local dump
  files after restore, and never commit dumps or secrets.
- Object names must be preserved exactly. Postgres major version must match (16).
- Everything reversible: GCP resources stand up in parallel; if verification
  fails, tear them down and Replit is unaffected.

DELIVERABLES:
- Cloud SQL instance with a verified data copy (row counts recorded).
- GCS bucket with a verified object copy (counts + sample checksums recorded).
- scripts/migrate-objects.mjs (idempotent object copy) committed.
- Secrets in Secret Manager and a runtime service account with least-privilege
  IAM, ready for the Cloud Run deploy step.
