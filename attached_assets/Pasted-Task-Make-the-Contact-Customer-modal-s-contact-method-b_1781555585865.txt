Task: Make the Contact Customer modal's contact-method buttons more user-friendly, and turn "Send Email" into a real templated-email send (editable preview + confirmation) instead of a manual note.

Context — files involved:
- src/react/components/modals/ContactCustomerModal.tsx (the modal)
- server.js (existing contact-customer endpoints ~lines 6148–6300; deposit-invoice email-preview endpoint ~line 6386; deposit-invoice send/email pipeline ~line 6831 — copy these patterns)
- email-templates.js (add a new template to TEMPLATE_DEFS + SAMPLE_VARS)
- src/react/components/modals/ContactCustomerModal.stories.tsx and the relevant test/ suites

1) RELABEL THE THREE CONTACT-METHOD BUTTONS
In ContactCustomerModal.tsx the buttons render values from METHOD_LABEL ("Called"/"Emailed"/"WhatsApp"). DO NOT change METHOD_LABEL — it is also used to render the contact-history log lines, which must keep reading "Called/Emailed/WhatsApp". Instead add a button-only map:

  const METHOD_BUTTON_LABEL: Record<Method, string> = {
    call:     'Log Call',
    email:    'Send Email',
    whatsapp: 'Log WhatsApp',
  };

Use METHOD_BUTTON_LABEL[method] for the button label (keep the "✓ " prefix when logged, e.g. "✓ Send Email"). Leave the "+ log another" affordance as-is. Also change the modal title from `Call ${displayName}` to `Contact ${displayName}`.

2) "LOG CALL" AND "LOG WHATSAPP" — UNCHANGED
Keep the existing inline note-panel flow exactly: click → note textarea → Confirm → POST {method, note} to /api/card-actions/contact-customer/:contactId/attempts.

3) "SEND EMAIL" — NEW CONFIRM + EDITABLE-PREVIEW + TEMPLATED SEND
Clicking Send Email must NOT open the plain note panel. Instead open an email-send flow in the same inline panel area (new local state e.g. emailFlow: 'idle' | 'preview' | 'sending'):
  a. On click, fetch a rendered preview from a new endpoint POST /api/card-actions/contact-customer/:contactId/email-preview (mirror /api/card-actions/open-deal/deposit-invoice-email-preview at server.js ~6386). It loads the new template via getEmailTemplate('contact_customer_followup'), substitutes firstName (derive from contactName, fallback 'there'), and returns { subject, text, html } via renderEmail.
  b. Show the rendered subject in a single-line TextField and the body (the text) in a multiline TextField, both EDITABLE and pre-filled from the preview. Show recipient name + email (displayName / contactEmail) above them. If contactEmail is empty, disable the flow with a clear message.
  c. Buttons: Cancel (closes panel, no send) and Send Email (primary). On Send set emailFlow='sending', disable inputs, show a spinner.

4) NEW SEND ENDPOINT (sends email AND logs the attempt atomically)
Add POST /api/card-actions/contact-customer/:contactId/send-email in server.js, isAuthenticated + requirePrivilege('member') (same level as /attempts). Body: { subject, body }. It must:
  a. Validate contactId (/^\d+$/) and that subject/body are non-empty; fetch the contact's email from HubSpot (reuse the contact-properties fetch the contact-customer loader at server.js ~6030 already uses). 400 if no email on file.
  b. Send via the existing pipeline used by the deposit-invoice flow (server.js ~6831): _createMailTransport(), _buildFromHeader(), _buildReplyTo(), then transport.sendMail({ from, replyTo?, to: contactEmail, subject, text: body, html: body wrapped as escaped <p> lines }). If the transport isn't configured, return 500 with a clear message and do NOT log the attempt.
  c. On successful send, in the SAME transaction, log the email attempt exactly like /attempts does: set email_sent = TRUE (upsert contact_attempt_tracking) and insert into contact_attempt_log with method='email' and an auto-note: Follow-up email sent: "<subject>". Return the same response shape as /attempts (call_attempted, email_sent, whatsapp_sent, attempted_at, attemptLog) so the frontend updates identically.
On the frontend, after a successful send reuse the success handling from handleConfirmAttempt (set emailSent, attemptLog, lastAttemptAt/lastAttemptBy, call broadcastContactAttemptLogged(contactId)), close the email panel, show a brief "Email sent" confirmation. Error handling mirrors handleConfirmAttempt (400 → message, 500 → "Something went wrong on our end." with retry).
