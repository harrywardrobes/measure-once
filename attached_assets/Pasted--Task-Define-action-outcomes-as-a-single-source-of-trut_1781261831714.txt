# Task: Define action outcomes as a single source of truth, surface them on the Workflow page, and update the skill

## Context

Every card action handler (contact_customer, arrange_visit, design_visit_followup, open_deal, etc.) already has outcomes in practice, but they are scattered and implicit:

- Server routes hardcode inline maps — e.g. `OUTCOME_MAP` in `POST /api/card-actions/arrange-visit/outcome`, `OUTCOME_STATUS` in `POST /api/card-actions/design-visit-followup/outcome`, `TARGET_MAP` in `POST /api/card-actions/contact-customer/:contactId/advance-status` (all in `server.js`).
- Modals hardcode their own outcome buttons/labels (e.g. `ContactCustomerModal.tsx` has Called / Emailed / WhatsApp toggles and Send upload link / No response terminal paths).
- The Workflow tab (`src/react/pages/admin/WorkflowPage.tsx` → `ModalDetailCard`) only shows free-text prose from `HANDLER_MODAL_SUMMARY` in `src/react/utils/handlerMeta.ts`.

Nothing enforces that these three agree. We want one typed registry of outcomes per handler type that the server validates against, the modals render from, and the Workflow page displays — so they can never drift.

There are two kinds of outcome and the distinction must be modelled explicitly:
- **Terminal outcomes** — finish the action and (usually) move the card to a new lead status. E.g. contact_customer: "Send upload link", "No response".
- **Partial outcomes** — log progress but leave the card where it is. E.g. contact_customer: "Called", "Emailed", "WhatsApped".

## Phase 1 — Audit, then build the shared outcome registry

1. Audit every handler type's real outcomes by reading the modal component (paths in `HANDLER_COMPONENT_META` in `src/react/utils/handlerMeta.ts`) **and** its server route(s) in `server.js` / `photo-reviews.js`. The registry must describe what the code actually does — do not invent outcomes. Starting inventory to verify against code:
   - `contact_customer` — partial: `call` (Called), `email` (Emailed), `whatsapp` (WhatsApped) → no status change, logged to `contact_attempt_tracking`/`contact_attempt_log`; terminal: `send_upload_link` (hands off to upload_photos_and_info → AWAITING_PHOTOS), `attempted_to_contact` → ATTEMPTED_TO_CONTACT, `no_response` → NO_RESPONSE.
   - `arrange_visit` — terminal: `booked`, `email_sent`, `not_proceeding`; resulting status varies by visitType (see `OUTCOME_MAP` in server.js — preserve the design/survey variants exactly).
   - `design_visit_followup` — terminal: `confirmed` → DESIGN_SCHEDULED, `invite_resent` → DESIGN_INVITED, `not_proceeding` → NOT_SUITABLE.
   - `review_customer_photos` — terminal: not suitable → NOT_SUITABLE, rough estimate → ROUGH_ESTIMATE.
   - `upload_photos_and_info` — terminal: link sent → AWAITING_PHOTOS.
   - `open_deal` — partial: amend (stays OPEN_DEAL); terminal: accept → DEPOSIT_INVOICE, decline → DECLINED_DEAL.
   - `deposit_invoice_followup` — partial: re-send invoice, send reminder, log call, amend deal (all stay DEPOSIT_INVOICE); terminal: arrange survey (status change — verify), not proceeding → DECLINED_DEAL.
   - `schedule_visit`, `summarise_phone_call`, `start_design_visit`, `show_message` — verify in code; show_message has no outcomes (empty list is valid).

2. Create the registry so **both** the CommonJS server and the React/TS client can import it. Follow the existing shared-module pattern (`src/react/utils/resolveActionLabel.mjs` is the precedent for client/test sharing; server.js uses `require`, so the data module itself must be plain JS the server can load — e.g. `shared/handler-outcomes.js` (CJS) — with a typed re-export in `handlerMeta.ts`). Shape per outcome:

   ```ts
   interface ActionOutcome {
     key: string;                  // canonical key — MUST equal the value the API accepts today
     label: string;                // user-facing label, e.g. "No response"
     kind: 'terminal' | 'partial';
     setsLeadStatus: string | null;            // hs_lead_status written; null = no status change
     setsSubstatus?: string | null;            // hw_lead_substatus where relevant (arrange_visit)
     variants?: Record<string, { setsLeadStatus: string; setsSubstatus?: string | null }>; // e.g. arrange_visit design/survey
     description?: string;         // one short line, e.g. "Logged to attempt history; card stays put"
   }
   ```

3. Re-export it from `src/react/utils/handlerMeta.ts` as
   `export const HANDLER_OUTCOMES: Record<HandlerType, ActionOutcome[]> = …`
   so `scripts/check-handler-meta.mjs` (run via `npm run test:handler-meta`) automatically enforces that every handler type defines its outcomes. This CI check discovers any `Record<HandlerType, …>` table in that file — no script changes needed, but confirm it passes.

## Phase 2 — Server consumes the registry

Replace the inline outcome maps in `server.js` with lookups derived from the shared registry:
- `arrange-visit/outcome` `OUTCOME_MAP`, `design-visit-followup/outcome` `OUTCOME_STATUS`, `contact-customer/:contactId/advance-status` `TARGET_MAP`, and the equivalent validation in open_deal / deposit-invoice / photo-review outcome routes.
- Build the validation sets and status writes from the registry (terminal outcomes only where appropriate). Error messages should list valid keys derived from the registry.
- **Zero behaviour change**: same accepted keys, same statuses written, same error semantics. The existing test suites (`test/arrange-visit`, `test/design-visit-followup`, `test/contact-attempt-history`, `test/deposit-invoice-followup`, `test/photo-reviews`, `test/card-action-handlers`) must pass unmodified — if a test needs changing, the refactor changed behaviour: stop and fix the refactor instead.

## Phase 3 — Modals consume the registry

In each modal, replace hardcoded outcome keys and button labels with imports from the registry (keys at minimum; labels where the current UI text matches — where a modal's button text intentionally differs from the canonical label, keep the UI text but reference the canonical key). Do not redesign any modal flow. Storybook stories for affected modals must still render.

## Phase 4 — Workflow page display

In `ModalDetailCard` (WorkflowPage.tsx), add two rows below the existing Steps/HubSpot/Emails rows, rendered from `HANDLER_OUTCOMES`:
- **Outcomes:** terminal outcomes as chips, each showing `Label → RESULTING_STATUS` (use the variants record for arrange_visit, e.g. "Booked → DESIGN_SCHEDULED / SURVEY_SCHEDULED"), or `Label → no status change` when `setsLeadStatus` is null.
- **Progress logged:** partial outcomes as a second, visually distinct chip row (these don't move the card).
- Omit a row when the handler has no outcomes of that kind; show nothing extra for show_message.
- Style consistently with the existing chip rows (small chips, caption labels). Tooltips may carry `description`.
- Once outcomes render from the registry, trim any now-redundant outcome prose from `HANDLER_MODAL_SUMMARY.steps`/`hubspot` strings so the card doesn't say everything twice.

- No DB schema changes; the registry is code, not data.
- Keep WorkflowPage.tsx from growing unwieldy — extract the outcomes rows into a small component if needed.
