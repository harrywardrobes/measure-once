# Feature: "Connect your services" modal (reuses the navbar connection-status backend)

## Goal
Add a single, reusable modal that lets a user connect/reconnect the app's external
integrations (Google Calendar, QuickBooks, HubSpot, and any future service). It must:

1. Auto-open ONCE PER SESSION whenever ANY tracked service is in an error state
   (e.g. a Google Calendar auth/connection error), so the user can fix the connection
   inline WITHOUT backing out of whatever modal/flow they're currently in.
2. Be re-openable at any time by clicking the service-status icons in the top navbar.
3. Be driven entirely by the EXISTING connection-status backend used by the navbar
   icons â€” do NOT create a second source of truth or duplicate status polling.
4. Be extensible: adding a new service in ONE place must make it appear in BOTH the
   navbar icon row AND this modal automatically.

    * `ServiceStatus` = 'ok' | 'error' | 'warning' | 'checking'

## Phase 1 â€” Extract a shared service registry (single source of truth)
Create `src/react/lib/connectionServices.ts` (or `.tsx` if it imports MUI icons) that exports
ONE registry describing every connectable service. Move the label/Icon data currently inline
in GlobalHeader's `SERVICE_CONFIG` here, and EXTEND each entry with connect metadata:

    export interface ServiceDescriptor {
      key: ConnectionService;
      label: string;
      Icon: React.ComponentType<{ fontSize?: 'small' | 'medium' }>;
      // How the user fixes/establishes this connection:
      //  - 'oauth'      â†’ navigate to `connectUrl`
      //  - 'admin-only' â†’ only admins can connect (e.g. QuickBooks)
      //  - 'managed'    â†’ configured by an admin via secrets/settings, not a per-user
      //                   OAuth click (e.g. HubSpot) â†’ show guidance + link to settings
      //  - 'status-only'â†’ never user-connectable (e.g. database) â†’ omit from the modal list
      connect: 'oauth' | 'admin-only' | 'managed' | 'status-only';
      connectUrl?: string;        // for oauth / admin-only
      disconnectUrl?: string;     // optional POST endpoint
      settingsHref?: string;      // for 'managed' (e.g. admin HubSpot settings page) if one exists
      helpText: string;           // one line shown under the row
    }

    export const SERVICE_DESCRIPTORS: ServiceDescriptor[] = [

      { key: 'hubspot',    label: 'HubSpot',         Icon: HubIcon,     connect: 'managed',

        helpText: 'Core app database.' },

Then refactor `GlobalHeader.tsx` to build its `SERVICE_CONFIG`/`SERVICE_KEYS` from this
registry (keep behavior identical) so there is no duplicated label/Icon data. Verify the
header still renders all four icons exactly as before.

DESIGN PRINCIPLE TO HONOR: the modal's service list = `SERVICE_DESCRIPTORS` filtered to
`connect !== 'status-only'`. Adding a future service later = add ONE descriptor entry (plus
the existing `ConnectionService` union + the existing /api/<svc>/status check) and it shows
up in both the navbar and the modal with zero extra UI code.

## Phase 2 â€” Build `ConnectServicesModal`
Create `src/react/components/modals/ConnectServicesModal.tsx`. Use MUI `Dialog` (it renders in
a portal at high z-index, so it correctly stacks ON TOP of any modal the user already has open
â€” this is the whole point: the user fixes the connection without leaving their current flow).

Props: `{ open: boolean; onClose: () => void; highlightService?: ConnectionService }`.

Behavior/layout:
- Title: "Connect your services" with a subtitle explaining these integrations keep the app
  working.
- Body: map over the registry (excluding 'status-only'). For each service render a row with:
    * the service Icon + label,
    * a status chip derived from `useServiceStatuses().get(key)` â€” reuse the SAME status â†’
      colour/label mapping the navbar uses (factor `statusLabel`/`statusBadgeColor` out of
      GlobalHeader into the shared lib if helpful, rather than copy-pasting), with states
      Connected (ok) / Disconnected (error) / Degraded (warning) / Checking,
    * the right action depending on `connect`:
        - 'oauth':      "Connect" button â†’ `window.location.href = connectUrl` when not ok;
                        "Disconnect" (POST disconnectUrl) when ok.
        - 'admin-only': same as oauth IF `isAdmin` (usePrivilege); if NOT admin, show a
                        disabled state with "Ask an admin to connect QuickBooks".
        - 'managed':    no Connect button; if `settingsHref` exists and user is admin, an
                        "Open settings" link, else a short "Managed by an admin" note.
    * `helpText` under the row.
- Visually emphasize `highlightService` (e.g. subtle highlighted border/background on that row)
  so when the modal auto-opens from a specific error, the user's eye goes to the right service.
- A "Done" / close button. `onClose` on backdrop click + Esc.

Constraints (match repo conventions â€” there are lint checks that WILL fail otherwise):
- Use MUI `sx` only; NO inline `style={{...}}`, NO raw hex colours in the component (use
  existing design tokens / `BRAND_COLORS` / theme palette). See existing modals in
  src/react/components/modals/ for the house style.
- Add `data-testid` attributes (e.g. `connect-services-modal`, `connect-row-google`,
  `connect-action-google`) consistent with existing testid patterns.

## Phase 3 â€” Auto-open (once per session) + reopen from navbar
Create a small controller so a SINGLE modal instance is mounted app-wide and both triggers
feed it. Prefer extending the existing connection singleton rather than adding new globals:

- Add module-level state to `ConnectionToastContext.tsx` (alongside `_lastKnown`) for
  "connect modal" open state + which service triggered it, plus `useConnectModal()` hook and
  an imperative `openConnectModal(service?)` exposed on the existing `window.__connectionToast`
  bridge (so vanilla-JS callers can open it too).
- AUTO-OPEN RULE: subscribe to status changes (the existing `_updateCallbacks`/notify path).
  When ANY service transitions INTO 'error' (not 'warning', not 'checking'), AND the modal
  hasn't already been auto-opened this session, open it with `highlightService` = the first
  service in error. Persist the "already auto-opened" flag in `sessionStorage` (key e.g.
  `mo:connectModalShownThisSession`) so it shows at most once per browser session. Manual
  reopen (below) must always work regardless of that flag.
- REOPEN: make the navbar "Service status" icon group in GlobalHeader clickable â€” wrap the
  row (or each `ServiceStatusBadge`) so clicking calls `openConnectModal()` (or
  `openConnectModal(svc)` to highlight that service). Keep it accessible: it should be a real
  button/role with an aria-label like "Manage service connections", keep the existing tooltips.

## Phase 4 â€” Mount + edge cases
- Mount ONE `<ConnectServicesModal>` driven by `useConnectModal()` high in the authenticated
  tree so it overlays everything. Good candidates: inside `ConnectionToastProvider` or near
  where GlobalHeader mounts in src/react/main.tsx. Ensure it is NOT mounted on the suppressed
  pages already listed in ConnectionToastContext (login, set-password, onboarding,
  design-visit/* sign-off).
- Offline: when `useOnlineStatus()` is false, do NOT auto-open (every check fails offline and
  it'd be noise) â€” match how the navbar swaps icons for the OfflinePill.
- Do not change any server routes or auth gating. QuickBooks stays admin-gated server-side;
  the UI must mirror that, not bypass it.

## Acceptance criteria (summary)
- One modal, driven by the SAME status backend as the navbar icons (no duplicated polling/config).
- Auto-opens at most once per session on any service error, highlighting the offending service,
  layered on top of any open modal so the user never loses their place.
- Reopenable by clicking the navbar service-status icons.
- Adding a future service = one descriptor entry + one /api/<svc>/status check; it appears in
  both navbar and modal with no further UI work.
- Respects admin gating, offline state, and the suppressed-page list. typecheck + lint + tests green.
