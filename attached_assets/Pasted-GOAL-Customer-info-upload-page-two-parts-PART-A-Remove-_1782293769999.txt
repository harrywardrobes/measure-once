GOAL
Customer-info upload page, two parts.

PART A — Remove email/phone corrections for EXISTING contacts (ready-now).
  When the page opens for someone we already have on file, the customer must NOT
  be able to change their email or phone. Those edits are done manually in HubSpot.

PART B — Brand-new customer creation via the customer-info page (net-new).
  When the page opens for a brand-new customer with NO existing HubSpot contact,
  email + phone are REQUIRED and are used to CREATE the HubSpot contact with a
  null/empty lead status. Postcode is NOT required for this. Reuse the existing
  contact-creation backend rather than rebuilding it.

REUSABLE BACKEND (verified)
  POST /api/contacts (server.js ~line 2008) already creates a HubSpot contact from
  firstname/lastname/email/phone (+ address), derives a customer_number, and
  returns the contact. It's used by the staff "New customer" dialog
  (NewCustomerDialog in src/react/pages/CustomersPage.tsx ~line 1856, submit ~1949).
  Reuse this logic — but note three mismatches that must be handled:
   1. AUTH: /api/contacts is staff-only (isAuthenticated + requirePrivilege('member')
      + requireHubspotToken). The customer-info page is PUBLIC (token-based, no
      login). You cannot call the staff endpoint directly from the public page.
      => Extract the core "create HubSpot contact" logic into a shared helper
         (e.g. createHubspotContact({ firstname,lastname,email,phone,postcode?,
         leadStatus })) and call it from BOTH the staff endpoint and the new
         public customer-info submit path. Keep the staff endpoint's auth intact;
         the public path stays gated by the existing customer-info bearer token.
   2. LEAD STATUS: the endpoint hardcodes hs_lead_status:'OPEN_DEAL' (server.js
      ~2034 assertLeadStatusKey + ~2047). The helper must take leadStatus as a
      parameter. Public customer-info creation passes NULL/empty; the staff dialog
      keeps OPEN_DEAL (do not change staff behaviour unless told).
   3. POSTCODE: currently required — the backend derives the customer_number area
      prefix from the postcode and returns 400 if zip is missing (server.js
      ~2025–2031, fallback prefix 'XX' already exists but the 400 fires first).
      Make postcode OPTIONAL for the new-customer creation path: when absent, skip
      the 400 and use the 'XX' fallback prefix for the customer_number.

PREREQUISITE DESIGN DECISION (call out before building Part B)
  Today the customer-info page can't be opened without an existing contact:
  customer_info_submissions.contact_id is TEXT NOT NULL
  (migrations/1749200000005_customer-info.js) and every link-creation path fetches
  an existing HubSpot contact (e.g. POST /api/customer-info/by-contact/:contactId/
  generate-link, customer-info.js ~627). Decide and state how a brand-new customer
  reaches a contactless page — recommended: a staff-generated "new customer" link
  that creates a submission row with no contact_id yet (requires making contact_id
  nullable via migration, and creating + back-filling the contact on submit). Do
  NOT silently reuse the existing per-contact link flow. If this entry point is out
  of scope, stop and confirm before changing the schema.
