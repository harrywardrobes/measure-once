# Task: "Priority first" sort on the Customers page should order by last-contacted, with an admin toggle

## Context (current behaviour)
On the Customers page, the sort dropdown option **"Priority first"** currently does NOT
sort by last-contacted. It pins contacts with no lead status to the top and then orders
the rest by **newest created date**. This logic is duplicated in two places and must stay
in sync:
- Server: `server.js` ~line 1757 (the `priorityFirst` comparator in the `/api/contacts-all`
  handler).
- Client (offline mirror): `src/react/hooks/usePaginatedContacts.ts` — `offlineComparator`
  (case `'priority'`, ~line 107) and the `effectiveComparator` block (~line 186).
- The dropdown is defined in `src/react/pages/CustomersPage.tsx` (`SORT_OPTIONS`, ~line 138);
  `priorityFirst` is passed through as `sortBy === 'priority'` (~line 1083 and 316).

The "small time indicator" on each customer card is the relative last-contact time. It is
rendered from `lastAttempt.at` (see `src/react/utils/activityTooltip.tsx`), with a fallback
to the HubSpot `notes_last_contacted` property. IMPORTANT GOTCHA: `lastAttempt` is fetched
SEPARATELY and PER-PAGE via `POST /api/contacts/urgency` (server.js ~line 2953, which reads
a contact-attempts table ordered by `attempted_at DESC`). It is NOT part of the main
`/api/contacts-all` payload that the sort runs over. So sorting the full list by it requires
making a last-contacted timestamp available for ALL contacts BEFORE pagination — you cannot
just read it off the already-loaded contact objects.

## Required behaviour
Change "Priority first" so it sorts customers by **last-contacted, longest-since-contact
first** (i.e. the most overdue-for-follow-up customers rise to the top). Specifically:
1. Customers with **no contact history at all** (no last-attempt timestamp and no
   `notes_last_contacted`) are treated as **highest priority** and sorted to the **TOP**.
2. Remaining customers are ordered by their last-contacted timestamp **ascending**
   (oldest contact first / most recently contacted last).
3. Use the same "last contacted" value the card's time indicator uses: prefer the
   contact-attempt timestamp (`lastAttempt.at` equivalent), fall back to
   `notes_last_contacted`. Keep server and offline-client comparators consistent.
4. Decide and document a stable tie-breaker (e.g. createdate desc) so ordering is
   deterministic.

For the server sort to work across the whole list, source a last-contacted timestamp for
every contact (e.g. query the contact-attempts table for the latest `attempted_at` per
`hubspot_contact_id` and coalesce with `notes_last_contacted`) and sort on that before
paginating. Keep the existing per-page urgency fetch working for the card display.

## Admin-configurable setting
Make this behaviour switchable from the admin panel, using the EXISTING page-filter-config
infrastructure (do not invent a new mechanism):
- Add a new key to `PAGE_FILTER_CONFIG_DEFAULTS` in `server.js` (~line 4934), e.g.
  `customers_priority_sort_mode` with values like `'last_contacted'` (new default) vs
  `'newest'` (legacy behaviour) — or a boolean if you prefer. Migration/seed follows the
  existing `page_filter_config` pattern (see `migrations/1749200000002_server-misc.js` ~line 217).
- Expose it through the existing endpoints: it's already returned by
  `GET /api/page-filter-config` and `GET/PATCH /api/admin/page-filter-config`
  (server.js ~lines 4977–5039) once added to defaults; confirm the PATCH allow-list picks it up.
- Render a control for it in the admin Settings page `src/react/pages/admin/SettingsPage.tsx`,
  in the existing **Customers** section (near `customers_page_size`, ~line 336). Include a
  short helper/label explaining that "Priority first" sorts by last-contacted
  (longest-since-contact first), with never-contacted customers on top.
- Have both the server `priorityFirst` comparator and the client `offlineComparator` read
  this setting and apply the last-contacted ordering when enabled, falling back to the old
  newest-first behaviour when not.
