# Task: Clean up unused card-action handlers and merge duplicate scheduling actions

## Context
The dashboard has 13 card-action handler types (see `CARD_ACTION_HANDLER_CONFIG_VALIDATORS`
in `server.js`, ~line 5320, and the `HandlerType` union in
`src/react/components/CardActionModalsHost.tsx`). Three of them are unused, and two of
them are duplicates of each other sharing the exact same modal and backend functions.

Do this work in two phases. **Do not start Phase 2 until Phase 1's check passes.**

## Phase 1 — Verify against the live database (do this FIRST)
Query the database:

```sql
SELECT h.id, h.name, h.type, h.config, b.stage_key, b.status_key, b.substatus_id
FROM card_action_handlers h
LEFT JOIN card_action_handler_bindings b ON b.handler_id = h.id
WHERE h.type IN ('schedule_visit', 'schedule_delivery_window',
                 'schedule_installation_slot', 'add_design_visit_to_calendar');
```

- If any rows exist for `schedule_delivery_window` or `schedule_installation_slot`,
  STOP and report them to me before deleting anything.
- Rows for `schedule_visit` or `add_design_visit_to_calendar` are fine — they get
  migrated in Phase 2, not lost. Report what you found, then continue.

## Phase 2a — Fully remove `schedule_delivery_window` and `schedule_installation_slot`
These can't be in active use: handler bindings only support the sales / designvisit /
survey stages, and Delivery/Installation stages are out of the dashboard's scope.
Remove every trace:

- `server.js`: their entries in `CARD_ACTION_HANDLER_CONFIG_VALIDATORS` (the
  `CARD_ACTION_HANDLER_TYPES` set derives from its keys, so it updates automatically)
- `src/react/components/CardActionModalsHost.tsx`: their `case` branches and their
  members of the `ModalState`/`HandlerType` union
- Delete `src/react/components/modals/DeliveryWindowModal.tsx` and
  `src/react/components/modals/InstallationSlotModal.tsx` (they are ~95% copy-paste
  duplicates of each other anyway)
- `src/react/utils/handlerMeta.ts`: remove their keys from ALL `Record<HandlerType, …>`
  tables (`HANDLER_MODAL_SUMMARY`, `HANDLER_TYPE_LABELS`, `HANDLER_EMAIL_TEMPLATES`,
  `HANDLER_COMPONENT_META`)
- Admin UI: their config blocks in `src/react/pages/admin/HandlerConfigBlocks.tsx`,
  plus `DeliveryWindowConfig.stories.tsx`, `InstallationSlotConfig.stories.tsx`, and
  references in `ActionHandlersPage.tsx`, `WorkflowPage.tsx`,
  `ConflictResolverModal.stories.tsx`, `HandlerConfigBlocks.test.tsx`,
  `HandlerConfigBlocks.mdx`, `_HandlerConfigBlockStoryHelpers.tsx`
- `src/react/components/modals/SchedulingModals.stories.tsx` and `.mdx`
- Legacy vanilla-JS paths: any branches/labels in `public/card-action-handlers.js`
  and `public/admin.html`
- Tests: update `test/card-action-handlers/run.js` and
  `test/scheduling-past-time-guard/run.js` to drop these types
- Add a DB migration that deletes any `card_action_handlers` rows (and cascading
  bindings) with these two types, as a safety net.

Full delete — do NOT keep the modals dormant. Git history preserves them if the
Delivery/Installation stages ever go live.

## Phase 2b — Merge `schedule_visit` and `add_design_visit_to_calendar` into ONE action
These two are the same function registered twice: both open `ScheduleVisitModal`, both
create the calendar event via `POST /api/events`, both optionally send the
`visit_confirmation` email. The only difference is that `add_design_visit_to_calendar`
locks the visit type to "design".

Replace both with a single handler type `schedule_visit` whose **config sub-type**
determines the grouping in the back end:

- Config schema: `{ visitType: 'design' | 'survey' | 'other', defaultDurationMin?, defaultTitle? }`
  with `visitType` REQUIRED (validate in `CARD_ACTION_HANDLER_CONFIG_VALIDATORS`).
  Drop the now-unused `installation`, `remedial`, `workshop` options from
  `VALID_VISIT_TYPES`.
- `ScheduleVisitModal` always renders locked to the configured `visitType` — the
  sub-action is chosen by the admin when binding the handler to a slot, not by the
  user at click time.
- Admin UI (`HandlerConfigBlocks.tsx` / `ActionHandlersPage.tsx`): one "Schedule visit"
  action with a required visit-type sub-option selector. Update its label/summary in
  `handlerMeta.ts` accordingly.
- **DB migration**: convert existing rows —
  `UPDATE card_action_handlers SET type = 'schedule_visit',
   config = config || '{"visitType":"design"}' WHERE type = 'add_design_visit_to_calendar';`
  and set `visitType: 'design'` as a default on any existing `schedule_visit` rows
  missing it. Existing bindings must keep working unchanged.
- Remove `add_design_visit_to_calendar` everywhere `schedule_delivery_window` was
  removed in Phase 2a (validators, ModalState union, handlerMeta tables, admin UI,
  vanilla-JS paths, tests, stories such as `ScheduleVisitConfig.stories.tsx` /
  `ScheduleVisitModal.stories.tsx`).
