Implement three changes to the "Arrange visit" flow in this repo. Visits are stored as Google
Calendar events on a shared calendar (the `visits` DB table is retired), tagged with
`extendedProperties.private.moContactId` and `moVisitType`. Relevant code:

- Modal:    src/react/components/modals/ArrangeVisitModal.tsx
            (steps: loading → call → booked/email → done; "Confirm booking" = handleBooked)
- Calendar: GET  /api/events?contactId=<id>  (server.js ~2480) — returns that contact's
            FUTURE events only (timeMin=now, singleEvents, orderBy=startTime), filtered by
            privateExtendedProperty moContactId=<id>
            POST /api/events (server.js ~2512) — creates an event; pass moContactId/moVisitType
            to tag it; the rest of the body is the Google event resource (summary, start, end,
            location, description, etc.)
- Outcome:  POST /api/card-actions/arrange-visit/outcome (server.js ~5994) — patches HubSpot
            lead status. booked → DESIGN_SCHEDULED / SURVEY_SCHEDULED.
            Visit type comes from the contact's lead status in
            POST /api/card-actions/arrange-visit.

=== 1. Prevent duplicate visits ===
When the user confirms a booking (handleBooked), first check whether that customer already has
a visit booked — even at a different day/time. Use GET /api/events?contactId=<contactId> to
look up the contact's existing FUTURE calendar events. ANY upcoming visit counts as a
duplicate, regardless of visit type (design or survey).

If one exists, do NOT book immediately. Show a confirmation dialog (reuse the existing
confirm-dialog pattern such as DiscardConfirmDialog — not window.confirm):
  Message: "<Customer name> already has a visit booked for <day + date> at <HH:MM>."
           (e.g. "Adam Smith already has a visit booked for Tuesday 18th at 16:00")
           Format the existing event's start in the user's locale; if there are several,
           reference the soonest upcoming one.
  Buttons: "Cancel"            (dismiss, do not book, stay in the modal)
           "Confirm Schedule"  (proceed and book anyway)

On "Confirm Schedule", continue the normal booking flow (address PATCH → arrange-visit/outcome
→ calendar event creation) and LEAVE the existing visit in place — the customer ends up with
both visits booked (do not cancel or modify the existing event). If there is no existing visit,
book without prompting, as today.

=== 2. Confirm a "sent but not confirmed" visit on open ===
A visit whose proposal was emailed but not yet confirmed sits in a "sent" state (the email_sent
outcome). When the Arrange-visit modal opens for such a contact, surface that unconfirmed visit
and offer a "Confirm appointment" action. Confirming it sets the contact's lead_status to
SURVEY_SCHEDULED (for survey-type contacts) or DESIGN_SCHEDULED (for design-type) — set ONLY
the parent lead_status; do not change any substatus. Detect the sent/unconfirmed state from the
contact info returned by POST /api/card-actions/arrange-visit (extend that endpoint's response
if needed to expose it). Show this confirm affordance by default whenever a visit is sent but
not confirmed.

=== 3. Calendar event location + description ===
Wherever the app builds the POST /api/events body for a visit, set:
  - event.location    = the customer's address (the structuredAddress collected in the modal,
                        rendered via formatAddress())
  - event.description = the visit notes
Apply this on the booking path.

Add/extend tests alongside the existing ones (test/arrange-visit/, test/profile-google-calendar/)
covering: duplicate detected → prompt shown → Cancel aborts / Confirm Schedule books and keeps
both; confirming a sent visit sets the parent lead_status with no substatus change; and event
location/description are populated from address/notes.
