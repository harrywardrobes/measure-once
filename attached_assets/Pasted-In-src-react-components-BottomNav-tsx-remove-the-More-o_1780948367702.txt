In src/react/components/BottomNav.tsx, remove the "More" overflow button when there are few enough nav tabs that overflow isn't needed, and fix the layout-shift/jank that happens while the nav loads.

1. Remove "More" when it isn't needed
Currently BAR_SIZE = 3 (line ~78) is hardcoded, so exactly 3 tabs always render directly and everything else — even a single item — gets pushed into the "More" overflow menu (rendered unconditionally at lines ~319-339, with no check for whether overflowItems actually has anything in it). With the Calendar tab removed, NAV (lines ~69-75) now has 4 visible items for most users (home, customers, projects, invoices for managers; fewer for non-managers).

Change the logic so that when the total number of currently-visible nav items (visibleNav.length, lines ~151-155) is small enough to fit directly (e.g. 4 or fewer), render all of them as primary tabs and skip the "More" button entirely — bypass BAR_SIZE/resolvedPrimaryKeys/role-based overflow splitting in that case.
Only fall back to the existing primary/overflow split (with the "More" button) when visibleNav.length exceeds that threshold.
Also fix the existing bug where "More" renders even when overflowItems is empty — it should never show with zero items, regardless of the above change.
2. Fix "weird loading movements" (layout shift while the nav resolves)
The set of tabs shown can change after first paint because:

configLoaded/prefsLoading (from usePrefs(), line ~138, and loadRoleNavConfig(), lines ~192-213) start with DEFAULT_PRIMARY_KEYS (line ~77) as a fallback, then swap to the resolved role/pref-based set once the async fetch completes — changing barItems/overflowItems (and thus visible tabs/labels/icons) mid-render.
isManager (from usePrivilege(), line ~137) resolves asynchronously and a separate effect (lines ~221-226) re-applies defaults once it changes, which can again alter visibleNav after the initial paint.
Because nav items have no fixed per-item width (MUI BottomNavigationAction defaults + actionSx, lines ~246-258), each change in item count reflows/redistributes the row width, causing visible jank.
To fix this:

Render a stable, skeleton/placeholder version of the bottom nav (correct height, item count, and layout) immediately based on the best-known synchronous state (e.g. DEFAULT_PRIMARY_KEYS filtered by whatever privilege info is already available), rather than popping items in/out as configLoaded/isManager/prefs resolve.
Where possible, resolve isManager/role config/prefs before the first paint of the nav (e.g. block on a combined loading state) rather than rendering once with defaults and re-rendering once real data arrives — a single, correct render is better than a render-then-correct flow.
Give each nav item a fixed/min width (rather than letting BottomNavigationAction distribute width based on count) so that even if the item count does change, existing items don't visibly shift position — new items should appear without moving the ones already there.
Test with slow network conditions (throttle in devtools) to confirm there's no visible "jump" in tab count, position, label, or icon between first paint and final state, for both manager and non-manager accounts.
