In the "Arrange Visit" modal (opened from the customers page), the UI shows the warning "No phone number on record for this contact." even when the contact has a mobile number on file but no landline phone value — and when both numbers exist, only one should be shown but ideally both should be visible.

This is because server.js (around line 6174, in the POST /api/card-actions/arrange-visit handler) only reads the contact's phone HubSpot property:

const contactPhone = String(props.phone || '');
It never looks at mobilephone, so contacts that only have a mobile number (e.g. notifications@github.com, which has a mobilephone value but no phone value) trigger the false "no phone number" warning in ArrangeVisitModal.tsx (line ~305/338, where phone = contactInfo?.contactPhone || '').

Fix:

In server.js, confirm the request that fetches props for this endpoint includes mobilephone in the requested properties (similar to params: { properties: 'phone,mobilephone' } used elsewhere, e.g. around server.js line 6343) — add it if missing.
Instead of returning a single contactPhone string, return both numbers separately, e.g.:
const contactPhone = String(props.phone || '');
const contactMobilePhone = String(props.mobilephone || '');
res.json({ visitType, contactName, contactPhone, contactMobilePhone, contactEmail, contactAddress });
In ArrangeVisitModal.tsx, update the component to read both contactInfo?.contactPhone and contactInfo?.contactMobilePhone:
If neither is present, show the existing warning "No phone number on record for this contact."
If only one is present, display that single number (labeled appropriately, e.g. "Phone" or "Mobile").
If both are present, display both, each labeled (e.g. "Phone: 020 1234 5678" and "Mobile: 07123 456789"), so the user can choose which to call/text.
Follow the labeling convention already used elsewhere in the codebase for distinguishing phone vs mobilephone (see CustomerDetailPage.tsx and AdminTeamPage.tsx, which map mobilephone → label 'mobile').

After the change, opening the visit invite for a contact with only a mobile number should display that number (no false warning), and a contact with both a landline and mobile number should display both, each clearly labeled.
