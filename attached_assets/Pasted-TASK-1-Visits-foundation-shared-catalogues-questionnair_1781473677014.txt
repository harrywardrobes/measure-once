TASK 1 — "Visits" foundation (shared catalogues + questionnaire engine + Design Visit re-org)
Goal: turn the Design-Visit-specific admin area into a shared "Visits" foundation that both Design and (later) Survey visits build on. No loss of existing functionality.

1a. Neutral shared catalogue tables (data migration)
Create new node-pg-migrate migration(s) that create neutral, shared catalogue tables and migrate existing data into them, then repoint all foreign keys and code:

catalog_handles — id, name, style, image_url, supplier_name, supplier_code, price_pence (nullable), notes, sort_order, created_at, updated_at
catalog_doors — id, name, image_url, colour (nullable), finish (nullable), supplier_name, supplier_code, notes, sort_order, created_at, updated_at
catalog_finishes — id, name, material_type (e.g. board/laminate/veneer/paint), image_url, supplier_name, supplier_code, notes, sort_order, created_at, updated_at
catalog_ranges — furniture ranges: id, name, description, sort_order, created_at, updated_at
catalog_pairings — admin-curated "these go well together": id, door_id FK→catalog_doors, handle_id FK→catalog_handles, finish_id FK→catalog_finishes (nullable), recommended (bool), notes, sort_order. Use this to suggest matching handles/finishes when a door is chosen.
Add any other catalogue you find is currently hard-coded or duplicated and would sensibly be shared (flag what you added and why).
Migration steps:

Migrate rows: design_visit_handles → catalog_handles, design_visit_door_styles → catalog_doors, design_visit_furniture_ranges → catalog_ranges (preserve ids/sort order/images).
Repoint FKs: design_visits.handle_id → catalog_handles, design_visits.furniture_range_id → catalog_ranges, design_visit_rooms.door_style_id → catalog_doors.
Replace the old design_visit_* catalogue admin + wizard endpoints with /api/admin/catalog/handles|doors|finishes|ranges|pairings (CRUD + image upload) and /api/catalog/... for in-wizard reads. Keep the old routes as thin aliases that 308/redirect or proxy to the new ones for one release, to avoid breaking anything mid-deploy; mark the aliases with a // TODO remove after migration comment.
Update all React references, types, and BroadcastChannel keys (e.g. catalog_handles_changed).
Drop the old catalogue tables only after data is confirmed migrated (do this in the same migration's up, with a correct down). Do not drop anything you haven't copied first.
1b. Shared questionnaire engine (used by Design Visit now, Survey later)
Tables: visit_questions (id, scope ['room'|'visit'], applies_to text[] e.g. {'design','survey'}, label, type ['yesno'|'choice'|'text'|'number'], options JSONB (for choice: list + single/multi flag), required bool, active bool, sort_order, created_at, updated_at) and visit_answers (id, visit_type, visit_id, room_id (nullable, for visit-level), question_id FK, answer JSONB, created_at).
Admin questionnaire builder UI (add/edit/reorder/remove, set scope, type, options, required, and which visit types it applies_to). Optional cheap conditional show-if logic only if low-effort.
Build a reusable <QuestionnaireRenderer> React component that renders a question set and captures answers, plus a server helper to load/save answers.
Wire it into the Design Visit wizard now: visit-level questions render in Step 1, per-room questions render per room in the rooms step; answers save with the design visit. Seed a few sensible starter questions but keep them all editable.
1c. Admin "Visits" tab re-org
Rename the admin "Design Visit" tab to "Visits" with sub-sections: Catalogues (handles, doors, finishes, ranges, pairings — shared), Questionnaire (the builder above), Design Visit (existing design-visit-specific settings: handler config, T&C, durations, statuses). Leave room for a future Survey Visit sub-section (Task 2 adds it).
Keep the existing T&C versioning (terms_conditions_versions) shared.
1d. Extract reusable components
Refactor the Design Visit wizard so the survey can reuse, without changing its behaviour: a generic wizard/drawer shell, a room editor card, the room-image uploader (pending-uploads + orphan recovery), catalogue dropdowns (handles/doors/finishes with pairing suggestions), and the <QuestionnaireRenderer>. Keep DesignVisitWizard working by composing these extracted pieces.
