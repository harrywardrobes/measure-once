# Follow-up: reusable QuickBooks payment history component

## Context
The deposit-invoice follow-up action flow now exists (`DepositInvoiceModal.tsx`,
`POST /api/card-actions/deposit-invoice`, payment-state banner derived from the
deposit invoice's Balance vs TotalAmt). This task extracts and generalises that
into a reusable payment history component showing real QuickBooks payment data
(date, amount, method, which invoice it paid), used in three places.

Relevant existing pieces:
- QuickBooks helpers in `quickbooks.js` (`fetchFromQuickBooks`, token handling,
  CustomerRef ownership checks). Payments come from the QuickBooks `Payment`
  entity, queryable by `CustomerRef`; each Payment's `Line[].LinkedTxn` links it
  to the invoice(s) it paid. Invoices for the customer are queryable by
  `CustomerRef` too (the open-deal loader already queries estimates this way).
- `qb_settings.payment_stages` defines named payment milestones (e.g. deposit,
  pre-delivery, final) — useful labels when relating payments to invoices.
- Frontend conventions: React + TypeScript + MUI, API wrappers in
  `src/react/utils/api.ts`, shared modal pieces in
  `src/react/components/modals/`.

## Task 1 — Shared backend endpoint
Add `GET /api/quickbooks/contacts/:contactId/payments` (same auth level as the
other per-contact QuickBooks reads, ownership-checked via CustomerRef):
- Query ALL QuickBooks payments for the customer, plus their invoices, and join
  them so each payment row has: payment date, amount, payment method (from
  `PaymentMethodRef`, resolved to its display name), reference/doc number, and
  the invoice(s) it was applied to (doc number + a label such as "Deposit" when
  the invoice matches `design_visits.deposit_invoice_id`, or a payment-stage
  name from `qb_settings.payment_stages` where identifiable).
- Also return per-invoice summary: total, balance, paid amount, and derived
  status (`paid` / `partial` / `unpaid`), and an overall summary (total invoiced,
  total paid, total outstanding).
- Return `qbConnected: false` cleanly when QuickBooks isn't connected rather
  than erroring. Cache responses briefly (in-memory, ~60s, keyed by contactId)
  to avoid hammering the QuickBooks API when several components mount.

## Task 2 — Self-fetching component with variants
Create `src/react/components/PaymentHistory.tsx`:

```ts
type PaymentHistoryProps = {
  contactId: string;
  variant: 'banner' | 'list';
  invoiceId?: string;   // optional: focus/filter on one invoice (e.g. the deposit invoice)
  title?: string;
};
```

- The component fetches its own data through a shared hook
  `usePaymentHistory(contactId)` (new file in `src/react/hooks/`) wrapping the
  endpoint above, with loading skeleton, error state, and a "QuickBooks not
  connected" state linking to the QuickBooks admin settings page.
- **`banner` variant**: one compact line/chip row. When `invoiceId` is given:
  green "Paid — £X on {date} ({method})", amber "Partially paid — £X of £Y",
  or amber "Awaiting payment — £Y due". Without `invoiceId`: overall summary
  ("£X paid of £Y invoiced, £Z outstanding"). This replaces the bespoke banner
  built inside DepositInvoiceModal in the previous task — refactor that modal
  to use this component and delete the now-duplicated logic, keeping the
  behaviour where the modal's option highlighting depends on the deposit
  invoice's paid state (expose what the modal needs from the hook, not by
  re-fetching).
- **`list` variant**: a small MUI table — date, amount, method, reference,
  invoice applied to (with its label, e.g. "Deposit — INV-1042") — plus the
  per-invoice summary above it. Empty state: "No payments recorded in
  QuickBooks yet."
- Both variants must work anywhere (modal or page) without the host passing
  data in.

## Task 3 — Use it in three places
1. **DepositInvoiceModal** — `variant="banner"` with `invoiceId` set to the
   deposit invoice (refactor as described above).
2. **CustomerDetailPage.tsx** — add a "Payments" section using
   `variant="list"`, placed consistently with the page's existing sections.
   Only render it for customers who have QuickBooks activity (hide the section
   cleanly when there are no invoices and QuickBooks is connected; show the
   not-connected hint to admins only).
3. **OpenDealActionModal** — `variant="banner"` (no invoiceId) shown in the
   hub step, so staff see any existing payments/outstanding balance before
   accepting or amending a deal. Keep it unobtrusive — a single line above the
   options.

## Conventions
- Don't introduce a data-fetching library; follow the existing fetch-in-hook
  patterns used elsewhere in `src/react/hooks/`.
- Currency formatting consistent with the rest of the app (GBP).
- All QuickBooks reads ownership-checked; no payment data exposed for contacts
  the user can't access.
- TypeScript types exported for the endpoint payload so all three hosts share
  them.

## Testing
Integration tests for the payments endpoint: connected with payments, connected
with none, disconnected, ownership rejection, and correct paid/partial/unpaid
derivation. Component-level: banner states (paid/partial/unpaid/summary) and
list empty state. Manually verify all three placements render and that
DepositInvoiceModal still highlights options correctly after the refactor.

When done, summarise files changed and confirm the old in-modal banner logic
was removed rather than duplicated.
