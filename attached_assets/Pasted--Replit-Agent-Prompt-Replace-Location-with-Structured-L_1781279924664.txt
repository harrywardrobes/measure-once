# Replit Agent Prompt: Replace "Location" with Structured, Localized Addresses

## Goal

Replace every generic "location" / free-text address field in this app with a single, structured, localization-friendly address model used consistently across the database, API, UI forms, and rendered output. **HubSpot remains the single source of truth for the customer's address** — the app must read from and write to HubSpot's standard contact properties.

Do not change unrelated behaviour (lead status sync, invoicing, photo reviews, auth). Work incrementally and keep the app running between steps.

## 1. Canonical address model

Define one shared address type and zod schema (put it in `shared/` so both server and React code import the same definition):

```ts
interface StructuredAddress {
  addressLines: string[];      // min 1, max 5 — street, building, unit/suite
  locality: string;            // city / town / village
  administrativeArea: string;  // state / province / region / county (optional in practice)
  postalCode: string;
  countryCode: string;         // ISO 3166-1 alpha-2, e.g. "GB", "US", "JP"
}
```

Rules:
- This exact shape is used in **all** JSON payloads, DTO validation, and new DB storage. No endpoint may accept or return a flat `location` string for an address going forward.
- Default `countryCode` is `"GB"` (this is a UK business).
- Validation: `addressLines` must contain 1–5 non-empty strings; `locality` and `postalCode` required; `administrativeArea` optional; `countryCode` must be a valid 2-letter ISO code. **Do not apply strict regional regexes to `postalCode`** — trim and length-check only, since formats vary by country.
- Build two pure, unit-tested utilities in `shared/`:
  - `formatAddress(addr, { multiline?: boolean })` — renders in the country's envelope order:
    - Western (default, e.g. GB/US): lines → locality → administrativeArea → postalCode → country
    - Eastern (JP, CN, KR, TW): country → postalCode → administrativeArea → locality → lines
    - Omit the country name when `countryCode === "GB"` (home market). Omit empty components. Single-line mode joins with `", "`.
  - `hubspotToAddress(props)` / `addressToHubspot(addr)` — see §2.

## 2. HubSpot mapping (source of truth)

The app currently reads/writes only `address`, `city`, `zip` on HubSpot contacts. Extend to the full standard property set — **do not create custom HubSpot properties**:

| App field            | HubSpot contact property | Notes |
|----------------------|--------------------------|-------|
| `addressLines`       | `address`                | Join lines with `"\n"` (newline) when writing; split on newlines when reading (first segment = line 1, rest = subsequent lines). HubSpot's street address field supports multi-line values. |
| `locality`           | `city`                   | |
| `administrativeArea` | `state`                  | HubSpot labels this "State/Region". |
| `postalCode`         | `zip`                    | |
| `countryCode`        | `country`                | HubSpot stores a free-text country **name**. Write the English country name (e.g. "United Kingdom"); when reading, map the name back to an ISO code via a lookup table, falling back to `"GB"` if blank/unrecognised. |

Update every HubSpot property list and payload in `server.js` accordingly:
- `GET /api/contacts/:id` (~line 2039): add `state,country` to the `properties` query; return a `structuredAddress` object built with `hubspotToAddress()` alongside the raw properties.
- `PATCH /api/contacts/:id` (~line 2060): replace the flat `address`/`city`/`zip` whitelist entries with acceptance of a `structuredAddress` object (validated by the shared zod schema), converted via `addressToHubspot()` before patching HubSpot.
- `POST /api/contacts` (~lines 1967–1986): accept the full `StructuredAddress` instead of a bare `postcode`; map via `addressToHubspot()`.
- All four card-action blocks that assemble `contactAddress` from `[address, city, zip]` (~lines 5853–5866, 6196–6207, 6300–6310, 6379–6389): fetch the full property set and use `formatAddress()` instead of manual joins. Keep the response field name `contactAddress` for display, but also include the structured object as `contactStructuredAddress` so modals can edit components.

## 3. Database migrations (node-pg-migrate, in `migrations/`)

1. **`customer_info_submissions`** (currently `address_line1`, `city`, `postcode` TEXT columns): add a `structured_address JSONB` column matching the 5-key schema. Backfill existing rows: `addressLines: [address_line1]`, `locality: city`, `postalCode: postcode`, `administrativeArea: ""`, `countryCode: "GB"`. Keep the legacy columns for now (read fallback), but all new writes go to `structured_address`.
2. **`design_visits`** (currently `location TEXT`): add `structured_address JSONB`. Backfill existing rows as `addressLines: [location]`, `countryCode: "GB"`, other fields empty. Keep `location` populated with `formatAddress(..., { multiline: false })` on write so old reads/exports don't break.
3. Every migration must have a working `down`.

## 4. API changes

- `POST /api/customer-info/:token` (customer-info.js ~line 773): replace `addressLine1`/`city`/`postcode` body fields with a `structuredAddress` object validated by the shared zod schema. Store it in the new JSONB column (and mirror into the legacy columns for compatibility). **New behaviour:** after a successful submission, also PATCH the submitted address to the customer's HubSpot contact via `addressToHubspot()` — the DB row remains as an audit record, but HubSpot must end up holding the submitted address. Reuse the existing HubSpot retry/rate-limit handling.
- `GET /api/customer-info/by-contact/:contactId` (~line 1281): return `structuredAddress` (fall back to assembling it from legacy columns for old rows).
- Design-visit create/update endpoints (design-visits.js): accept `structuredAddress` instead of the free-text `location` string; persist to both new and legacy columns as described in §3.

## 6. Google Places Autocomplete

- Add Google Places Autocomplete to `<AddressInput>`: a search box above the fields that, on selection, populates all 5 components from the place's `address_components` (street_number + route → addressLines, postal_town/locality → locality, administrative_area_level_1/2 → administrativeArea, postal_code → postalCode, country short_name → countryCode).
- API key comes from a Replit secret `GOOGLE_PLACES_API_KEY`; load the Places library lazily only when an `<AddressInput>` mounts. If the key is missing or the script fails to load, the component must degrade gracefully to manual entry — autocomplete is an enhancement, never a requirement.
- Manual editing of every field stays possible after autocomplete fills them.
- Autocomplete and map features read their runtime configuration from the admin settings described in §7 — no redeploy needed to change behaviour.

- Persist settings in the existing `app_settings` key/value table (a single `google_maps_settings` JSON value is fine), exposed via admin-only `GET`/`PUT /api/admin/google-maps-settings` endpoints with zod validation. The client loads them through the existing app-bootstrap/settings flow so `<AddressInput>` and map previews honour changes at runtime.

Settings to include (as many as practical — group into cards):

1. **Connection & health**
   - Status indicator showing whether the `GOOGLE_PLACES_API_KEY` secret is configured (show only last 4 characters; never expose the key to the browser).
   - "Test connection" button: runs a server-side round-trip (Autocomplete query → Place Details lookup) and reports success/failure, latency, and the raw error message on failure.
   - Per-API status checks: Places Autocomplete, Place Details, Geocoding, Maps Static/JavaScript — show which respond with the current key, so a key missing an enabled API is obvious.
2. **Master switch** — enable/disable all Google Maps features app-wide. When off, every `<AddressInput>` silently falls back to manual entry.
3. **Autocomplete behaviour**
   - Country restriction list for suggestions (default: GB only; allow adding more).
   - Results language (default `en-GB`).
   - Result type filter (addresses only vs. addresses + establishments).
   - Debounce in ms and minimum characters before querying (cost controls).
   - Session-token usage toggle (on by default — keeps Google billing per-session rather than per-keystroke).
4. **Visibility (per-surface toggles)** — independently enable autocomplete and map previews for each surface: public customer-info form, design-visit wizard step 1, arrange-visit modal, staff contact editing. This controls where Google Maps features appear throughout the application.
5. **Map previews** — toggle showing a static map thumbnail/pin next to rendered addresses (ModalContactHeader, customer detail, visit modals), with zoom level and map type (roadmap/satellite) settings.
6. **Usage & diagnostics**
   - Server-side counters of autocomplete/details/geocode requests (today / this month) so spend is visible without leaving the app.
   - Recent errors list (last 20 failures with timestamp, surface, and error code).
   - A warning banner on this tab — and a subtle one for admins elsewhere — when the key is missing/invalid or the recent error rate is high.
   - Link out to the Google Cloud Console credentials page.
7. **Fallback behaviour** — choose between silent manual entry vs. showing users an unobtrusive "address lookup unavailable" notice when the API is down.

## 8. Acceptance criteria

1. No user-facing form anywhere collects an address as a single free-text "Location" field.
2. Editing an address in the app updates the HubSpot contact's `address`, `city`, `state`, `zip`, `country` properties; reloading the page shows the same data read back from HubSpot.
3. A customer-info form submission appears on the HubSpot contact record.
4. Starting a design visit pre-fills the visit address from the contact; overriding it does **not** change the contact's HubSpot address.
5. A Japanese (`JP`) address renders country → postal code → region → locality → lines; a GB address renders lines → city → county → postcode with no country line.
6. Labels switch correctly: GB shows "Postcode", US shows "ZIP Code", others show "Postal Code".
7. Existing rows (old submissions, old design visits) still display correctly via the backfill/fallback.
8. Unit tests cover `formatAddress`, `hubspotToAddress`/`addressToHubspot` (including round-trip of multi-line addresses and country name↔code mapping), and the customer-info submission → HubSpot push. Existing test suites still pass.
9. Admin → Developer shows a "Google Maps" sub-tab; "Test connection" returns a live success/failure result; settings persist across reloads.
10. Turning the master switch off (or a per-surface toggle) immediately removes autocomplete/map previews from the affected forms without a redeploy, and manual entry keeps working.
11. The API key value is never sent to the browser — only its configured/last-4 status.
