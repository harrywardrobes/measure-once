Add an admin-editable email template system to this app. Currently, all email subject lines, bodies, and footer text are hardcoded inline string literals across customer-info.js (e.g. sendCustomerInviteEmail, sendAdminNotificationEmail, sendCustomerThankYouEmail), auth.js (sendSetPasswordEmail), and photo-reviews.js. There is no DB-backed templating — I want to make these editable from the admin panel.

Requirements:

Database: Create a new email_templates table (raw pg, follow the existing pattern of ensureXTable() functions called at startup, e.g. ensureResendLogTable). Columns should include: id, key (unique identifier like customer_invite, admin_notification, customer_thank_you, set_password_resend, set_password_reset, photo_review_notification, etc.), subject (text, supports placeholders), body_text, body_html, footer_text, updated_at, updated_by.
Seed/fallback: On startup, seed the table with the current hardcoded subject/body/HTML/footer values (extract them from the three send functions listed above) so nothing breaks if a row is missing — fall back to the hardcoded default if no DB row exists for a key.
Template variables: Support placeholder substitution (e.g. {{customerName}}, {{address}}, {{link}}) — document which variables are available per template key, and substitute them when the email is actually sent. Keep this simple (basic string replace is fine, no need for a templating engine).
Backend routes: Add GET /api/admin/email-templates, GET /api/admin/email-templates/:key, and PATCH /api/admin/email-templates/:key (admin-auth protected, follow the same auth/middleware conventions as the existing /api/admin/card-action-handlers... and /api/admin/conflict-digest-settings routes around server.js lines 5980-6190 and 253-365 in SettingsPage.tsx's API usage).
Refactor send functions: Update sendCustomerInviteEmail, sendAdminNotificationEmail, sendCustomerThankYouEmail, sendSetPasswordEmail, and the photo-review notification function to load subject/body/html/footer from the DB (with in-memory caching if appropriate) instead of using inline string literals, then apply variable substitution before sending via the existing nodemailer transports.
Admin UI: Add a new admin page (e.g. src/react/pages/admin/EmailTemplatesPage.tsx) listing all template keys with an editor modal to update subject, body (text/html), and footer — model the editor UI/UX on the existing ActionHandlersPage.tsx (Dialog-based editor, save/cancel, validation messages, loading/disabled states) and SettingsPage.tsx (save button with loading state, success/error feedback). Show available placeholder variables for each template key in the editor so the admin knows what they can use.
Navigation: Add a link/route to the new page in the existing admin navigation alongside Action Handlers and Settings.
Keep the implementation consistent with existing conventions (raw pg queries, no ORM, JSONB only where it's already used, same auth middleware, same React/Tailwind component patterns). Don't introduce new dependencies (e.g. no templating libraries) — basic string substitution is sufficient.
