# Fix: "No phone number on record" showing incorrectly in Arrange Visit modal

## Problem

The Arrange Visit modal (opened via the card-action on the customers page) shows the
warning "No phone number on record for this contact." even for contacts that do have a
phone number in HubSpot.

There are three related root causes:

### 1. Draft-restoration skips the API call (most common cause)

In `src/react/components/modals/ArrangeVisitModal.tsx`, the `useEffect` that fires when
`open` changes has an early-return path for when a draft exists in sessionStorage:

```javascript
const hasDraft = draft.step && draft.step !== 'loading' && draft.step !== 'done';
if (hasDraft) {
  setStep(draft.step as Step);
  // ...
  return;  // ← never calls the API, so contactInfo stays null
}
```

Because `contactInfo` stays `null`, both `contactInfo?.contactPhone` and
`contactInfo?.contactMobilePhone` evaluate to `''`, so the "No phone number" warning
fires even when the contact has a phone number.

**Fix:** when restoring from a draft, still fetch the contact info from the server.
The address/slot fields can be restored from the draft as before, but the contact
details (phone, name, email) should always come from the fresh API response.
Alternatively, store `contactPhone` and `contactMobilePhone` in the draft object and
restore them too — whichever approach is simpler.

### 2. WhatsApp phone number field never fetched

HubSpot has a dedicated `hs_whatsapp_phone_number` property for the WhatsApp number.
Neither the arrange-visit endpoint nor the contact detail endpoint currently fetch it.

### 3. `/api/contacts/:id` doesn't fetch `mobilephone`

In `server.js`, the `GET /api/contacts/:id` handler (around line 1971–1974) only
requests:

```
properties: 'firstname,lastname,email,phone,address,city,zip,customer_number,hs_lead_status,createdate'
```

`mobilephone` is missing. This means `contact.properties.mobilephone` is always
undefined on the customer detail page, so the WhatsApp button passes an empty phone
string to the WhatsApp modal for any contact whose number is stored in `mobilephone`
rather than `phone`.

---

## Changes required

### `server.js` — `POST /api/card-actions/arrange-visit` (around line 6163–6179)

1. Add `hs_whatsapp_phone_number` to the properties fetched from HubSpot:

```javascript
params: {
  properties: 'firstname,lastname,email,phone,mobilephone,hs_whatsapp_phone_number,address,city,zip,hs_lead_status'
}
```

2. Extract all three phone fields and return them:

```javascript
const contactPhone          = String(props.phone                    || '');
const contactMobilePhone    = String(props.mobilephone              || '');
const contactWhatsAppPhone  = String(props.hs_whatsapp_phone_number || '');

res.json({
  visitType,
  contactName,
  contactPhone,
  contactMobilePhone,
  contactWhatsAppPhone,
  contactEmail,
  contactAddress,
});
```

### `server.js` — `GET /api/contacts/:id` (around line 1971–1974)

Add `mobilephone` and `hs_whatsapp_phone_number` to the properties list:

```javascript
params: {
  properties: 'firstname,lastname,email,phone,mobilephone,hs_whatsapp_phone_number,address,city,zip,customer_number,hs_lead_status,hw_lead_substatus,createdate'
}
```

(`hw_lead_substatus` is also missing from this list and is needed by the
CustomerDetailHeader substatus pill — add it while you're here.)

### `src/react/components/modals/ArrangeVisitModal.tsx`

1. **Extend the `ContactInfo` interface** to include the new field:

```typescript
interface ContactInfo {
  visitType: 'design' | 'survey';
  contactName: string;
  contactPhone: string;
  contactMobilePhone: string;
  contactWhatsAppPhone: string;
  contactEmail: string;
  contactAddress: string;
}
```

2. **Fix the draft-restoration early-return** so it no longer skips the API call.
   Change the `useEffect` to always fetch the contact info when `open` becomes true.
   Restore the address/slot fields from the draft after the fetch resolves — the draft
   should not prevent the phone numbers from being loaded. The simplest approach:

   ```javascript
   // Always fetch contact info; restore draft fields after
   setStep('loading');
   POST('/api/card-actions/arrange-visit', { contactId: ctx.contactId })
     .then((data: unknown) => {
       const d = data as ContactInfo;
       setContactInfo(d);
       const hasDraft = draft.step && draft.step !== 'loading' && draft.step !== 'done';
       if (hasDraft) {
         setStep(draft.step as Step);
         if (draft.address) setAddress(draft.address);
         if (draft.bookedSlotIso) setBookedSlot(dayjs(draft.bookedSlotIso));
         if (draft.slotIso) setSlots(draft.slotIso.map(s => (s ? dayjs(s) : null)));
       } else {
         setAddress(d.contactAddress || '');
         setStep('call');
         saveDraft(key, { step: 'call', address: d.contactAddress || '', slotIso: [null, null, null], bookedSlotIso: null });
       }
     })
     .catch((e: Error) => {
       setLoadError(e.message || 'Could not load contact info.');
       setStep('call');
     });
   ```

3. **Read all three phone fields** in the render section:

```javascript
const landline  = contactInfo?.contactPhone         || '';
const mobile    = contactInfo?.contactMobilePhone   || '';
const waPhone   = contactInfo?.contactWhatsAppPhone || '';
```

4. **Show the "No phone number" warning only if all three are empty**:

```javascript
{(landline || mobile || waPhone) ? (
  <Box sx={{ display: 'flex', flexDirection: 'column', gap: 1 }}>
    {landline && (
      <Box>
        <Typography variant="body2" color="text.secondary">Phone</Typography>
        <Typography variant="h6" sx={{ fontWeight: 600 }}>{landline}</Typography>
      </Box>
    )}
    {mobile && (
      <Box>
        <Typography variant="body2" color="text.secondary">Mobile</Typography>
        <Typography variant="h6" sx={{ fontWeight: 600 }}>{mobile}</Typography>
      </Box>
    )}
    {waPhone && (
      <Box>
        <Typography variant="body2" color="text.secondary">WhatsApp</Typography>
        <Typography variant="h6" sx={{ fontWeight: 600 }}>{waPhone}</Typography>
      </Box>
    )}
  </Box>
) : (
  <Alert severity="warning">No phone number on record for this contact.</Alert>
)}
```

### `src/react/pages/customer-detail/types.ts` — `ContactProperties`

Add the two missing properties to the interface so TypeScript is happy:

```typescript
export interface ContactProperties {

```

### `src/react/pages/CustomerDetailPage.tsx` — WhatsApp phone prop (around line 780 & 805)

Update both places that pass `phone` to `WhatsAppHistory` and `WhatsAppModal` to fall
back through all three fields:

```typescript
phone={
  contact.properties.phone ||
  contact.properties.mobilephone ||
  contact.properties.hs_whatsapp_phone_number ||
  ''
}
```

---

## Expected outcome

- Opening the Arrange Visit modal for a contact with only a `phone` value → shows that
  number under "Phone".
- Opening it for a contact with only a `mobilephone` value → shows that number under
  "Mobile".
- Opening it for a contact with only `hs_whatsapp_phone_number` → shows that number
  under "WhatsApp".
- Opening it for a contact with multiple numbers → shows all present numbers, each
  labelled.
- Opening it a second time (draft restored from sessionStorage) → still loads and
  displays the phone numbers correctly.
- Opening it for a contact with no phone fields at all → shows the existing "No phone
  number on record for this contact." warning.
