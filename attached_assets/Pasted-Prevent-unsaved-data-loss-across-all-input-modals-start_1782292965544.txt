Prevent unsaved-data loss across all input modals, starting with the "Contact Customer" modal.

PROBLEM
When a user is part-way through logging a call, email, or WhatsApp message in the
Contact Customer modal (src/react/components/modals/ContactCustomerModal.tsx), they
can exit the modal — or close/refresh the browser tab — and silently lose everything
they typed. The modal calls `onClose()` directly (e.g. around lines 525, 550, and the
FullScreenModal `onClose` at ~606) with no "Discard changes?" confirmation.

GOAL
1. No accidental data loss when a modal containing user input is closed by ANY route:
   - clicking Cancel / the close (X) button
   - clicking the backdrop / pressing Escape
   - closing, refreshing, or navigating away from the browser tab
2. Apply the SAME behaviour consistently to every modal that has a data-entry form.

REUSE EXISTING PATTERNS — DO NOT REINVENT
The repo already has the in-app guard built:
   - src/react/hooks/useDiscardGuard.ts  (shows "Discard changes?" when hasUnsavedChanges)
   - src/react/components/modals/DiscardConfirmDialog.tsx
These are already wired into ScheduleVisitModal, ArrangeVisitModal, PhoneSummaryModal,
GenericVisitEditModal, ReviewCustomerPhotosDrawer, ContactEditModal, and EmailTemplatesPage.
Follow those modals as the reference implementation.

The browser tab-close guard (`beforeunload`) currently only exists in
src/react/components/InvoiceDetailDrawer.tsx — it is NOT centralised.

WORK TO DO
A. Contact Customer modal (primary fix):
   - Compute an accurate `hasUnsavedChanges` for EACH mode (call / email / WhatsApp) —
     true when any field the user can edit differs from its initial/empty value.
     Make sure switching tabs within the modal is handled sensibly.
   - Route every close path through `useDiscardGuard` + `DiscardConfirmDialog`, exactly
     like ScheduleVisitModal does. Pass `isLocked` while a send/save is in flight so the
     guard doesn't fire mid-submit, and skip the prompt after a successful send (the
     existing auto-close timer paths around lines 489/504 must not trigger the prompt).

B. Tab-close / refresh protection:
   - Add a small reusable hook (e.g. useBeforeUnloadGuard(hasUnsavedChanges)) that
     registers a `beforeunload` listener only while there are unsaved changes, mirroring
     the logic already in InvoiceDetailDrawer. Refactor InvoiceDetailDrawer to use it too.
   - Use this hook in the Contact Customer modal and in every other input modal.

C. Apply consistently everywhere:
   - Audit every modal/drawer under src/react/components/modals (and ContactEditModal under
     pages/customer-detail) that contains a data-entry field. Any that lack the in-app
     discard guard and/or the beforeunload guard should get both.
   - Modals that are purely informational / read-only (no editable input) should be left
     as-is.

CONSTRAINTS
- Keep the existing visual design and copy of DiscardConfirmDialog; don't introduce a new
  confirmation UI.
- Don't prompt when there are genuinely no unsaved changes — closing an untouched modal
  should still close instantly.
- Add/extend unit tests in the style of the existing *.test.tsx modal tests to cover:
  dirty close shows the prompt, clean close does not, "Keep editing" returns to the form,
  and submit-in-flight does not prompt.

DELIVERABLE
Working changes plus tests, with the Contact Customer modal fully protected and the same
guard behaviour verified across all input modals.
