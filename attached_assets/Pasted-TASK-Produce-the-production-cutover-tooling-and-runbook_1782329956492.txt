TASK: Produce the production cutover tooling and runbook for moving Measure Once
from Replit to Cloud Run + Cloud SQL + GCS. Two committed deliverables:
(1) scripts/verify-migration.mjs — a read-only parity checker (row counts, object
counts, and migration-bookkeeping drift); and (2) docs/gcp-cutover.md — the
sequenced cutover runbook WITH a freeze window and a rollback plan. You do NOT run
gcloud or perform the cutover; you produce the script and the document. This is
the one step with real downtime and limited reversibility, so the runbook must be
precise and ordered.

DEPENDENCIES (must already be merged / done):
- Tasks #1–#4: RUN_MIGRATIONS_ON_BOOT flag, storage.js with the 'gcs' backend +
  list(), the production Dockerfile, scripts/migrate-objects.mjs, and the GCP
  stand-up from docs/gcp-migration.md (Cloud SQL, GCS bucket, service account,
  Secret Manager) all complete. GCP is standing ALONGSIDE Replit with a rehearsal
  data copy already verified.

CRITICAL PRE-FLIGHT RISK TO DOCUMENT PROMINENTLY:
- In production on Replit, node-pg-migrate was NEVER run at boot (migrations were
  skipped; Replit's publish-time diff owned the schema). So the pgmigrations
  bookkeeping table in the LIVE PROD database may NOT list all migrations that are
  actually reflected in the schema. If prod is restored into Cloud SQL and
  `npm run db:migrate` is then run, node-pg-migrate may try to RE-APPLY already-
  applied migrations and fail ("column/constraint already exists").
- The runbook MUST therefore start with a pgmigrations reconciliation check (see
  verify-migration.mjs below) and branch:
    * If pgmigrations already matches the migrations/ directory → restore, run
      `db:migrate` (no-op or applies only genuinely new files), proceed.
    * If pgmigrations is INCOMPLETE while the schema is current (the likely Replit
      case) → reconcile pgmigrations so it lists every already-applied migration
      as applied WITHOUT running them, and do NOT run pending migrations at
      cutover. Document exactly how (insert one pgmigrations row per already-
      applied migration file, run_on = now), and verify afterwards that
      `db:migrate` reports zero pending. Keep RUN_MIGRATIONS_ON_BOOT unset on Cloud
      Run; apply future migrations via the pre-deploy `npm run db:migrate` step.

DELIVERABLE 1 — scripts/verify-migration.mjs
────────────────────────────────────────────────────────────────────────────────
A READ-ONLY checker (no writes, no deletes) used during the rehearsal and the
real cutover. Given env SOURCE_DATABASE_URL, DEST_DATABASE_URL, and DEST_GCS_BUCKET:
1. DB row-count parity: for a fixed list of key tables (users, sessions,
   password_set_tokens, design_visits, customer_info_submissions, photo_reviews,
   quickbooks token table, pgmigrations, public.migrations — adjust to the real
   table names), SELECT count(*) on both SOURCE and DEST and report matches /
   mismatches.
2. Object-count parity: count source objects via the storage.js abstraction
   (storage.list(), default/'replit' backend — assert STORAGE_BACKEND is unset or
   'replit') and count DEST objects via @google-cloud/storage
   (bucket.getFiles() over DEST_GCS_BUCKET, ADC). Report the two totals.
3. Migration-bookkeeping drift: list the files in the migrations/ directory and
   compare against the pgmigrations rows in DEST_DATABASE_URL. Report which
   migration files are NOT recorded as applied in DEST (this is what tells the
   operator whether reconciliation is needed).
Output a clear PASS/FAIL summary per section; exit non-zero if any DB table count
mismatches or any required env var is missing. No hardcoded connection strings,
bucket names, or credentials. Match the project's .mjs script conventions.
Add npm script: "verify:migration": "node scripts/verify-migration.mjs".

DELIVERABLE 2 — docs/gcp-cutover.md
────────────────────────────────────────────────────────────────────────────────
A sequenced runbook. Key facts to bake in:
- The production domain is preserved (handle.harrywardrobes.co.uk). Because the
  domain does not change, the Google OAuth and QuickBooks redirect URIs stay the
  same — no OAuth client reconfiguration is needed. APP_URL stays the same too.
- Sessions live in Postgres (connect-pg-simple). If the sessions table is included
  in the final dump/restore, existing users stay logged in across cutover.
- Cloud Run may run multiple instances, so migrations are applied as a PRE-DEPLOY
  one-off (`npm run db:migrate` via the Cloud SQL Auth Proxy from a trusted
  shell), NOT via RUN_MIGRATIONS_ON_BOOT (leave it unset on Cloud Run).

Phases:

PHASE A — Build & push image (no traffic)
  Create an Artifact Registry repo; build the prompt-3 Dockerfile and push:
    gcloud artifacts repositories create measure-once --repository-format=docker \
      --location=$REGION
    gcloud builds submit --tag $REGION-docker.pkg.dev/$PROJECT_ID/measure-once/app:VERSION
  (or docker build + docker push to the same path).

PHASE B — Deploy to Cloud Run pointing at the REHEARSAL data (still no DNS)
  gcloud run deploy measure-once \
    --image $REGION-docker.pkg.dev/$PROJECT_ID/measure-once/app:VERSION \
    --region $REGION --port 8080 --allow-unauthenticated \
    --service-account measure-once-run@$PROJECT_ID.iam.gserviceaccount.com \
    --add-cloudsql-instances $PROJECT_ID:$REGION:$SQL_INSTANCE \
    --set-env-vars NODE_ENV=production,STORAGE_BACKEND=gcs,GCS_BUCKET=$GCS_BUCKET,APP_URL=https://handle.harrywardrobes.co.uk \
    --set-secrets DATABASE_URL=...:latest,SESSION_SECRET=...:latest,HUBSPOT_TOKEN=...:latest,<all other secrets> \
    --min-instances 1
  Note: DATABASE_URL for Cloud SQL via the connector uses a unix socket, e.g.
    postgres://app:PASS@/measureonce?host=/cloudsql/$PROJECT_ID:$REGION:$SQL_INSTANCE
  Smoke-test on the temporary *.run.app URL (login, a photo view from GCS, a
  customer-info submission). Use safe/sandbox SMTP/HubSpot/QB values until the
  real cutover so the rehearsal never emails customers or sends invoices.

PHASE C — Cutover window (brief planned downtime; pick an off-hours slot)
  1. Announce the maintenance window.
  2. FREEZE writes on Replit: take the Replit deployment offline (stop it / scale
     to zero) so no new rows or photos are written during the copy. (Full downtime
     is the simplest safe freeze; a read-only mode is not implemented and is out
     of scope.)
  3. Final DB migration: fresh `pg_dump` of the live prod DB → restore into a CLEAN
     Cloud SQL database (drop/recreate first so it's authoritative, not merged with
     the rehearsal). Then run the pgmigrations reconciliation/branch from the
     CRITICAL PRE-FLIGHT section and confirm `db:migrate` reports zero pending.
  4. Final object delta: re-run `DEST_GCS_BUCKET=$GCS_BUCKET npm run migrate:objects`
     — idempotent, so it copies only objects created since the rehearsal.
  5. Verify parity: run `npm run verify:migration` and confirm all DB table counts
     match, object totals match, and pgmigrations shows zero un-applied files.
  6. Point production at Cloud Run: create the Cloud Run domain mapping for
     handle.harrywardrobes.co.uk and update DNS to the records it provides
     (document TTL considerations — lower the DNS TTL a day BEFORE cutover to speed
     propagation). Switch the live secrets/integrations (SMTP/HubSpot/QB) from
     sandbox to production values if they were sandboxed in Phase B.
  7. Smoke-test the production domain end-to-end once DNS resolves to Cloud Run.

PHASE D — Post-cutover verification & soak
  Re-run verify:migration; watch Cloud Run logs/errors; confirm login, photo
  signing from GCS, email send, HubSpot sync, and a QuickBooks read all work in
  production. Keep the Replit deployment STOPPED but intact as a warm rollback for
  a defined soak period (e.g. 7 days) — do NOT decommission it in this task.

PHASE E — Rollback plan
  If a blocking problem appears DURING the window or shortly after: point DNS back
  to Replit and restart the Replit deployment. State the data-loss caveat plainly:
  any writes made against Cloud Run after cutover are NOT on Replit, so rollback is
  clean only within a short decision window before real production traffic writes
  new data. Define that decision window and who makes the call.

Include a SAFETY section: live customer PII (encrypt dumps, delete local dumps
after restore, never commit dumps/secrets); object names preserved exactly;
Postgres major version 16 on both sides; lower DNS TTL ahead of time; the cutover
is reversible only until DNS flips AND real writes land on Cloud Run.
