# Admin Configuration Panel — Workflow Tab & UI Consistency Improvements

## Goal

Make the Configuration section of the admin panel easier to understand and maintain. The admin should be able to see, at a glance, what happens at each action — which handler fires, what the modal flow does, what HubSpot status changes, and what email is sent — without needing to cross-reference multiple tabs manually.

## Decisions

- **New "Workflow" tab** added to the Configuration group (after Email Templates). This is the canonical reference view.
- **Accordion tree** replaces the ReactFlow chart. No ReactFlow dependency needed.
- **Inline summaries** added on related tabs so context is visible without switching tabs.
- **Live data refresh** — edits made on any tab are reflected immediately across all tabs (no manual reload).
- **Workflow map removed** from CardActionsPage — that collapsible section and its ReactFlow component are deleted.
- **Tab order unchanged** — only the new Workflow tab is added.

---

## Entities & Their Relationships (for context)

```
workflow.json (stages — read-only)
    └─ lead_status_config (MO stages / lead statuses — synced from HubSpot)
        └─ lead_substatuses (sub-statuses — pushed to HubSpot hw_lead_substatus)
            └─ card_action_handler_bindings
                └─ card_action_handlers (type, config)
                    ├─ modal flow (hard-coded per handler type)
                    ├─ HubSpot status change (configured in handler config)
                    └─ email_templates (rendered at send time)

stage_action_labels  →  action button label per stage × lead-status slot
```

- **Data sources (all read-only, no writes on this page):**
  - `GET /api/admin/lead-statuses` — lead statuses with stage, key, label, shorthand
  - `GET /api/admin/lead-substatuses` — sub-statuses with status_key, substatus_key, label, action_label
  - `GET /api/admin/card-action-handlers` — handlers with type, config, bindings (stage_key + status_key OR substatus_id)
  - `GET /api/admin/email-templates` — templates keyed by key
- Use `react-query` (or the existing fetching pattern) so data stays live.

#### 1.3 Accordion tree structure

Render a 3-level accordion using MUI `Accordion` / `AccordionSummary` / `AccordionDetails` (consistent with existing admin UI patterns):

```
▶ [Stage label]  (e.g., "Sales")
  ▶ [Lead Status label]  (e.g., "Awaiting Photos")
      Handler:    upload_photos_and_info  [chip]
      Modal:      2 steps — Confirm → Send customer link
      HubSpot:    Status → AWAITING_PHOTOS (no change) / Sub-status cleared
      Email:      customer_invite  [chip with link to Email Templates tab]
      Button:     "Send Photos Link"
      ────────────────────────────────
      ▶ [Sub-status label]  (e.g., "Photos Received")
          Handler:  review_customer_photos  [chip]
          Modal:    3 steps — View photos → Decide → Send email
          HubSpot:  Status → PHOTOS_RECEIVED
          Email:    not_suitable_email / rough_estimate_email  [chips]
          Button:   "Review Photos"
```

- Stages that have no card-actions bindings (Order, Workshop, Packing, etc.) are shown as collapsed, read-only rows with a label "No action handlers configured".
- Lead statuses with no handler show "No handler bound" in muted text.
- All chips for Handler and Email Template are clickable — they navigate to the corresponding tab (use `window.location.hash` or the existing tab navigation mechanism).

#### 1.4 Handler type → human-readable modal summary map

Create a static lookup in a shared `src/react/utils/handlerMeta.ts`:

```ts
export const HANDLER_MODAL_SUMMARY: Record<string, { steps: string; hubspot: string }> = {
  add_design_visit_to_calendar: { steps: '1 step — Pick date & time → create calendar event', hubspot: 'No status change' },
  schedule_visit:               { steps: '1 step — Pick date & time → create calendar event', hubspot: 'No status change' },
  summarise_phone_call:         { steps: '2 steps — Enter notes → AI summary review → save HubSpot note', hubspot: 'No status change' },
  show_message:                 { steps: '1 step — Display message popup', hubspot: 'No status change' },
  start_design_visit:           { steps: '3 steps — Enter details → select rooms → review → submit', hubspot: 'Status → config.submittedLeadStatus' },
  schedule_delivery_window:     { steps: '1 step — Pick window start & end → create calendar event', hubspot: 'No status change' },
  schedule_installation_slot:   { steps: '1 step — Pick slot date & duration → create calendar event', hubspot: 'No status change' },
  upload_photos_and_info:       { steps: '2 steps — Confirm → send customer link via email', hubspot: 'No status change' },
  review_customer_photos:       { steps: '3 steps — View submitted photos → decide → send outcome email', hubspot: 'Status → PHOTOS_RECEIVED or NOT_SUITABLE' },
  arrange_visit:                { steps: '3 steps — Call outcome → if booked: pick calendar slot → confirm', hubspot: 'Status → booked or callback sub-status' },
};
```

For `start_design_visit`, interpolate `config.submittedLeadStatus` from the handler's actual config at render time.

#### 1.5 Email template detection per handler

```ts
export const HANDLER_EMAIL_TEMPLATES: Record<string, string[]> = {
  upload_photos_and_info:  ['customer_invite'],
  review_customer_photos:  ['not_suitable_email', 'rough_estimate_email'],
  start_design_visit:      ['design_signoff_link'],

```

Render each template key as a chip. Clicking navigates to the Email Templates tab with that template pre-selected.
