DELIVERABLE 1 — scripts/migrate-objects.mjs
────────────────────────────────────────────────────────────────────────────────
Purpose: copy every object from the SOURCE bucket (the current/Replit backend,
read via storage.js) to a DESTINATION GCS bucket, preserving object names EXACTLY
(stored DB keys like obj:<id>.<ext> map to object names such as
design-visit-images/<id>.<ext> and the customer-info object paths — any rename
breaks resolution).

Design:
- SOURCE is read through the storage.js abstraction's DEFAULT backend
  (storage.list() + storage.downloadBytes()). The script must assert that
  STORAGE_BACKEND is unset or 'replit' and exit with an error otherwise (so it
  can never read and write the same GCS bucket).
- DESTINATION is GCS, accessed DIRECTLY in this script via @google-cloud/storage
  (already a dependency): `new Storage().bucket(process.env.DEST_GCS_BUCKET)`,
  Application Default Credentials only — no key files, no hardcoded bucket name.
- Enumerate ALL source objects via storage.list() (no prefix = everything).
  Support an optional --prefix=<p> flag to scope a run.
- For each object, bounded by a concurrency limit (env COPY_CONCURRENCY, default
  8):
    * Check the destination: if an object with the same name AND same byte size
      already exists (bucket.file(name).getMetadata()), SKIP it (this is what
      makes the script idempotent/resumable — re-runs only copy missing or
      size-mismatched objects).
    * Otherwise download bytes from source (storage.downloadBytes(name)) and
      upload to dest (bucket.file(name).save(buffer)); then verify the dest size
      matches the source byte length.
- Flags/env: --dry-run (log what WOULD copy, write nothing), --prefix=<p>,
  COPY_CONCURRENCY, DEST_GCS_BUCKET (required unless --dry-run).
- Output a final summary: total / copied / skipped / failed, list the failed
  object names, and exit non-zero if any failed.
- NEVER delete or mutate source objects. No DB access. No cutover.
- Match the project's existing .mjs script style (see scripts/gc-design-visit-
  photos.mjs for logging/flag conventions).

Operational note to include as a header comment: this script reads the SOURCE via
the Replit backend, so it must run in an environment where that bucket is wired
in (e.g. the Replit shell) WHILE GCP Application Default Credentials for the
destination are configured (GOOGLE_APPLICATION_CREDENTIALS or `gcloud auth
application-default login`). Document the exact invocation.

Add npm scripts:
  "migrate:objects":         "node scripts/migrate-objects.mjs"
  "migrate:objects:dry-run": "node scripts/migrate-objects.mjs --dry-run"

DELIVERABLE 2 — docs/gcp-migration.md
────────────────────────────────────────────────────────────────────────────────
A runbook a human executes. Use these PARAMETERS at the top (UK data residency —
we hold customer PII):
  PROJECT_ID=measure-once-prod   REGION=europe-west2 (London)
  SQL_INSTANCE=measure-once-pg   DB_NAME=measureonce   DB_USER=app
  GCS_BUCKET=measure-once-media-<unique-suffix>
Match Postgres MAJOR version to source: Replit uses Postgres 16 → Cloud SQL must
be POSTGRES_16.

Document these phases with the exact commands:

PHASE 0 — Project & APIs
  gcloud config set project $PROJECT_ID
  gcloud services enable sqladmin.googleapis.com storage.googleapis.com \
    secretmanager.googleapis.com run.googleapis.com \
    artifactregistry.googleapis.com compute.googleapis.com

PHASE 1 — Cloud SQL (Postgres 16)
  gcloud sql instances create $SQL_INSTANCE --database-version=POSTGRES_16 \
    --region=$REGION --tier=db-custom-1-3840 --storage-auto-increase \
    --backup --enable-point-in-time-recovery
  gcloud sql databases create $DB_NAME --instance=$SQL_INSTANCE
  gcloud sql users create $DB_USER --instance=$SQL_INSTANCE --password=<strong>
  (Prefer Cloud SQL Auth Proxy / private IP; store the password in Secret Manager,
   not in shell history.)

PHASE 2 — Cloud Storage bucket
  gcloud storage buckets create gs://$GCS_BUCKET --location=$REGION \
    --uniform-bucket-level-access --public-access-prevention
  (Object names must be preserved exactly — see the copy script.)

PHASE 3 — Service account & least-privilege IAM
  gcloud iam service-accounts create measure-once-run
  gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member="serviceAccount:measure-once-run@$PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/cloudsql.client"
  gcloud storage buckets add-iam-policy-binding gs://$GCS_BUCKET \
    --member="serviceAccount:measure-once-run@$PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/storage.objectAdmin"
  (App auth via ADC; no key files in the repo.)

PHASE 4 — Secret Manager
  Store each runtime secret (DATABASE_URL for Cloud SQL, SESSION_SECRET,
  HUBSPOT_TOKEN, GOOGLE_* OAuth, QB_*, GOOGLE_PLACES_API_KEY, the Google/QB token
  encryption keys). These get mounted into Cloud Run at cutover. Never commit
  values.

PHASE 5 — Database migration (Replit Postgres → Cloud SQL)
  pg_dump "$SOURCE_DATABASE_URL" --no-owner --no-privileges -Fc -f dump.pgc
  pg_restore --no-owner --no-privileges -d "$CLOUDSQL_DATABASE_URL" dump.pgc
  Note: a FULL pg_dump carries BOTH bookkeeping tables — pgmigrations
  (node-pg-migrate) AND public.migrations (the rate-limit package) — which must
  come across or the app re-runs migrations on first boot. Record and compare row
  counts for: users, sessions, design_visits, customer_info_submissions,
  quickbooks tokens, pgmigrations, public.migrations.

PHASE 6 — Object migration (→ GCS) — uses Deliverable 1
  Dry run first, then real copy, then verify:
    DEST_GCS_BUCKET=$GCS_BUCKET npm run migrate:objects:dry-run
    DEST_GCS_BUCKET=$GCS_BUCKET npm run migrate:objects
  Document the run environment (source bucket wired in + GCP ADC for dest), and
  the verification step: compare object COUNT and a sample of byte sizes/checksums
  between source and GCS. The script is idempotent — safe to re-run.

PHASE 7 — Verification (NO cutover)
  Optionally run the container (from the Dockerfile task) LOCALLY against Cloud SQL
  + GCS (STORAGE_BACKEND=gcs, GCS_BUCKET set, Cloud SQL via proxy) and smoke-test:
  log in, open a design visit with photos (signed URLs resolve from GCS), open a
  customer-info submission. Use safe/sandbox values for SMTP/HubSpot/QuickBooks so
  a test instance never emails customers or sends invoices. Leave Replit
  production untouched and authoritative.

Include a SAFETY section: source data is live customer PII — keep dumps encrypted,
delete local dump files after restore, never commit dumps or secrets; object names
must be preserved; Postgres major version must match (16); everything is
reversible (GCP stands up in parallel; tear down on failure and Replit is
unaffected). Note that the production cutover (DNS + Cloud Run + final delta sync +
RUN_MIGRATIONS_ON_BOOT decision) is a SEPARATE later step, not part of this task.
